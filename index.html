Rain (VA / PA) vs Card-Bridge vs DeCard

能力差异对比 — 把 Rain 拆成 VA / PA 两种模式，分别对位 DeCard 和 Card-Bridge
整理日期：2026-05-19

一、四方画像

Rain Managed

业界 Consumer Program 标杆 · rain-managed
API-first，用户分配智能合约账户， Rain代替客户管理卡组授权与清算，无PCI可查敏感信息
API 完整度★★★★★ 安全设计★★★★★ 运营友好度★★★★★

DeCard

DCS · Virtual Account Model (VA)
API-first + H5，用户分配EOA地址， DCS代替客户管理卡组授权与清算，敏感信息要强PCI认证（H5引导 or API）
API 完整度★★★★★ 安全设计★★★★★ 运营友好度★★★★★

Rain Partner Managed

业界 Partner-Managed 标杆 · partner-managed
API-first，Partner分配智能合约账户（商户自掌钱包），Partner 处理卡组授权与清算；无PCI可查敏感信息
API 完整度★★★★★ 安全设计★★★★★ 运营友好度★★★★★

Card-Bridge

DCS · Partner Account Model (PA)
API-first，无链上智能合约或者EOA，依靠Partner账户保证金，Partner 处理卡组授权与清算；强PCI认证，敏感信息要强PCI认证（H5引导 or API）
API 完整度★★★★★ 安全设计★★★★★ 运营友好度★★★★★

三、关键洞察

Rain 三大值得学

  • 双签名 + 平滑轮换
    Primary + Secondary 同时签发, 实现零停机轮换
  • 细节的运营能力更完善
  • 无 PCI 资质也能查 PAN/CVV

我们的明显差距

  • KYC 免活体：Rain 复用 Sumsub Share Token / API 直传都可免活体，DCS 两条路都必须加活体。
  • Webhook 密钥轮换：Card-Bridge 和 DeCard 都缺平滑切换机制（双签）。
  • 整体运营能力偏弱：比如错误码的定义不够明确、没有原始限额的设置、对账数据的明细等

我们的差异化优势

  • Card-Bridge虚转实可保留同 PAN，Rain 两种模式都不支持虚转实
  • Card-BridgeWebhook 交易金额 RSA 加密 + 加签，比 Rain 的 HMAC 签名更重的安全等级
  • DeCard提供实体卡物流通知最完整

四、场景导向 — 如果你最关心 ...

把卡密展示给用户，但我们没有 PCI 资质
Rain Managed / Rain Partner Managed 都是答案（密钥协商方案）。DCS 必须 合作方有 PCI 证书 才能API交互，或者只能H5页面的引导。
用户想从虚拟卡升级到实体卡，且保留同一张卡号
Card-Bridge 独有能力。Rain VA / PA 都只能新建一张卡
实体卡用户的"卡到哪了"体验
DeCard 最好（WebSocket + Webhook 双通道实时推送）； Card-Bridge 仅支持轮询查询（无推送）； Rain Managed / Rain PA 都不做物流。
商户用户已经在 Sumsub 完成 KYC，不想再做一次
四方都支持 Sumsub Share Token 复用；Rain Managed / Rain PA 可免活体，Card-Bridge & DCS（无论 Share Token 还是 API 直传）都必须加活体。
想做长期、稳定的密钥滚动而不停服
Rain Managed / Rain PA 都支持双签轮换。Card-Bridge / DeCard 密钥轮换都需要短暂窗口或人工协调。
想要"商户掌钱包、自己记账、自己决定授权"
Rain Partner Managed 是 Rain 这套生态里唯一答案；本质上和 Card-Bridge 的工作模型对位
想让 Rain/DCS 全包，商户接事件就行
Rain Managed 就是为此设计；和 DeCard 的角色定位相同 —— 但 Rain Managed 提供更完整交易事件、能查 PAN/CVV，明显比 DeCard 完整。
